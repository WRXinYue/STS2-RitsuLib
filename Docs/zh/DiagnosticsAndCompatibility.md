# 诊断与兼容层

本文专门说明 RitsuLib 在原版游戏外面补上的那一层“小型安全与兼容机制”。

重点包括：

- 帮助作者尽早发现错误的一次性警告
- 面向调试的缺失本地化兼容行为
- 原版系统本身不支持 Mod 内容时的窄桥接补丁

---

## 设计意图

RitsuLib 并不想把所有引擎问题都藏进隐式魔法里。

它更倾向于这条规则：

- 如果能尽早暴露真实错误，就尽早暴露
- 如果原版没有安全扩展点，框架可以补桥
- 如果某个 shim 会隐藏过多行为，就宁可保持显式

所以这层能力是刻意收敛、只处理边缘问题的。

---

## 一次性警告策略

RitsuLib 有几类诊断只会对同一个问题警告一次。

典型包括：

- 缺失资源路径
- 调试兼容模式下缺失本地化键

目的就是让日志保持“可行动”：

- 足够明显，能看到问题
- 又不会在每帧或每次打开界面时疯狂刷屏

---

## 资源路径诊断

显式资源 override 现在会通过 `AssetPathDiagnostics` 做校验。

当某个资源型覆写路径不存在时：

- 框架会输出一次警告
- 然后回退到原始资源路径或原始行为

这对角色资源尤其重要，因为原版游戏自己几乎没有可靠的角色资源兜底。

更详细的资源语义见 [资源配置与回退规则](AssetProfilesAndFallbacks.md)。

---

## 本地化 Debug 兼容模式

RitsuLib 提供了一个只用于调试阶段的 `LocTable` 缺失 Key 兼容层。

当开启 `debug_compatibility_mode` 后：

- `LocTable.GetLocString(key)` 缺失时返回占位 `LocString`
- `LocTable.GetRawText(key)` 缺失时直接返回该键本身
- 同一个缺失键只会警告一次

该模式默认关闭。

它的目的是降低调试本地化时的痛苦，不是替代正确的本地化内容。

Windows 下设置文件路径：

```text
%appdata%\SlayTheSpire2\steam\<user_id>\mod_data\com.ritsukage.sts2-RitsuLib\settings.json
```

## 注册冲突诊断

RitsuLib 会显式检查以下冲突：

- 模型 ID 冲突
- 纪元 ID 冲突
- 故事 ID 冲突

因为固定公开身份只有在冲突被明确报出来时才有意义。

常见触发场景包括：

- 同一个 Mod/类别下两个已注册模型的 CLR 类型名相同
- 两个纪元最终解析出了同一个 `Id`
- 两个故事最终解析出了同一个故事标识

一旦检测到冲突，框架会抛异常或输出错误日志，而不是静默接受模糊身份。

## Ancient 对话兼容层

RitsuLib 现在会在 `AncientDialogueSet.PopulateLocKeys` 之前，自动为已注册 Mod 角色追加基于本地化定义的 Ancient 对话。

这层能力的定位是“兼容便利层”，而不是替代作者控制：

- 对话 key 仍然由你自己编写
- 框架只是负责发现并追加，使 Mod 角色也能走与原版角色同一套 Ancient 对话模式

具体 key 结构见 [本地化与关键词](LocalizationAndKeywords.md)。

---

## 解锁兼容桥

原版的若干进度检查本来就是按原版角色设计的。

RitsuLib 通过一些较窄的桥接补丁，让你注册的解锁规则能在这些节点上对 Mod 角色生效，例如：

- 基于精英胜场的纪元解锁
- 基于 Boss 胜场的纪元解锁
- 进阶 1 的纪元解锁
- 局后角色解锁纪元
- 进阶显示解锁检查

这些补丁并不是在发明第二套进度系统，它们只是把你注册在 RitsuLib 里的规则，转发到原版原本会忽略 Mod 角色的检查点上。

规则本身的详细语义见 [时间线与解锁](TimelineAndUnlocks.md)。

---

## Freeze 异常本身就是一种诊断

当内容、时间线或解锁在冻结之后还被注册时，RitsuLib 会直接抛异常。

这应该被视为诊断机制，而不是框架“不够宽容”。

因为一旦晚注册，往往意味着：

- ModelDb 缓存可能已经建立
- 固定身份规则可能已经被使用
- 解锁过滤可能已经开始运行

此时最安全的做法就是尽早失败，而不是带着半初始化状态继续运行。

---

## 推荐的排查思路

当你看到某条兼容/诊断警告时，建议这样理解：

1. 先把它当作配置问题，而不是随机引擎不稳定
2. 缺失资源和缺失本地化，优先从源头修复
3. 调试兼容模式只在调试阶段使用
4. 如果已经有干净的显式 API，就不要反过来依赖兼容层

框架是在帮你“看见问题”，而不是帮你把问题永久藏起来。
